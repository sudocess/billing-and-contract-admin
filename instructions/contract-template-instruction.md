# Copilot Instructions — Contract HTML Template

## Overview

The contract output is a **multi-page A4 HTML document** generated by the dashboard's contract wizard. When the user reaches Step 7 (Generate) and clicks "Generate contract", the dashboard must build this HTML file dynamically by filling tokens from wizard state and injecting conditional blocks based on contract type, phase, and toggles.

The source template file is: `contract-template.html`
The filled output file is written to: `contracts/[CONTRACT_ID].html` (e.g. `contracts/2026-8832104-0002.html`)

Do not modify `contract-template.html` directly. Always generate a new filled file per contract.

---

## 1. Page Structure

The document is **not a single scrolling page**. It is divided into **multiple A4 page `<div>` blocks**, each representing one physical page (794px × 1123px at 96dpi, equivalent to A4 at 210 × 297mm).

### Page 1 — always contains:
- Full dark header (brand + contract ID)
- Title bar (project name + phase + contract type)
- Section 1: Parties
- Section 2: Project at a Glance
- Section 3: Scope of Work
- Section 4: Payment Schedule

### Page 2 — always contains:
- Slim continuation header (brand name + contract ID)
- Section 5: Revision Scope
- Section 6: Hosting & Add-ons
- Section 7: Technical Infrastructure & Third-Party Services

### Page 3 — always contains:
- Slim continuation header
- Section 8: Client Obligations
- Section 9: Intellectual Property
- Section 10: Confidentiality
- Section 11: Limitation of Liability
- Section 12: Cancellation & Termination
- Section 13: Governing Law
- Signatures block

Every page ends with a `.page-footer` showing `Page X of Y`.

If content on any page grows too long (e.g. many add-on rows, long deliverables), Copilot must split that page into two and update the total page count accordingly. The rule is: **never let `.page` content exceed 1123px in height**. Keep `min-height: 1123px` on each `.page` so short pages still fill A4.

### Page div structure

```html
<!-- PAGE 1 -->
<div class="page">
  <header class="contract-header"> ... </header>   <!-- page 1 only -->
  <div class="contract-title-bar"> ... </div>       <!-- page 1 only -->
  <div class="page-body"> ... sections ... </div>
  <div class="page-footer">
    <div class="pf-ref">Contract ID: [CONTRACT_ID] · Engaging UX Design</div>
    <div class="pf-num">Page 1 of 3</div>
  </div>
</div>

<!-- PAGE 2+ -->
<div class="page">
  <div class="page-header-cont">                    <!-- pages 2+ only -->
    <div class="phc-brand">Engaging UX Design — Service Agreement</div>
    <div class="phc-id">[CONTRACT_ID]</div>
  </div>
  <div class="page-body"> ... sections ... </div>
  <div class="page-footer">
    <div class="pf-ref">Contract ID: [CONTRACT_ID] · Engaging UX Design</div>
    <div class="pf-num">Page 2 of 3</div>
  </div>
</div>
```

## 3. Contract ID Format

Generate the contract ID automatically using this format:

```
[YEAR]-[CLIENT_CODE]-[PHASE_NUMBER]
```

- `YEAR` — current 4-digit year, e.g. `2026`
- `CLIENT_CODE` — 7-digit auto-generated internal code assigned to the client on first contract creation. Store this in the client record and reuse it on all subsequent contracts for the same client. Example: `8832104`
- `PHASE_NUMBER` — 4-digit zero-padded phase number starting at `0001`. Each new phase contract for the same client increments this. A scope extension or custom agreement uses the next available number.

Examples:
```
First contract for a new client:   2026-8832104-0001
Phase 2 for the same client:       2026-8832104-0002
Custom add-on for the same client: 2026-8832104-0003
```

## 4. Scope Phase Notes (`{{SCOPE_PHASE_NOTES}}`)

Insert a `.note` block below the deliverables paragraph describing what the selected phase covers. Use the prebuilt text below. Do not change the wording.

### Phase 1 — Strategy & Structure
```html
<div class="note">
  <strong>Phase 1 — Strategy &amp; Structure</strong> covers: project kick-off meeting, UX research and sitemap definition, content structure and wireframes, design direction proposal, and written Phase 1 sign-off by the client before Phase 2 begins.
</div>
```

### Phase 2 — Design & Prototype
```html
<div class="note">
  <strong>Phase 2 — Design &amp; Prototype</strong> covers: high-fidelity design in Figma, typography and layout refinement, a fully clickable interactive prototype, and client feedback rounds within the agreed revision scope before Phase 3 begins.
</div>
```

### Phase 3 — Build & Launch
```html
<div class="note">
  <strong>Phase 3 — Build &amp; Launch</strong> covers: full front-end and back-end development, integration of all dynamic applications, cross-browser and device testing, deployment to the live hosting environment, and final handover of all credentials and source code.
</div>
```

### Custom phase
```html
<div class="note">
  This is a Custom Agreement. The scope of work is as described above and agreed between both parties.
</div>
```

The user may also add extra deliverables notes from the dashboard. If they do, append them as a second `.note` block below the phase note.

## 5. Payment Rows (`{{PAYMENT_ROWS}}`)

### Standard project (Basic / Business / Enterprise) — 3-phase breakdown:
```html
<tr>
  <td>Phase 1 — 30%</td>
  <td>Due before Phase 1 work begins</td>
  <td class="amount" style="text-align:right;">[PHASE1_AMOUNT]</td>
</tr>
<tr>
  <td>Phase 2 — 40%</td>
  <td>Due after written client approval of Phase 2</td>
  <td class="amount" style="text-align:right;">[PHASE2_AMOUNT]</td>
</tr>
<tr>
  <td>Phase 3 — 30%</td>
  <td>Due before final delivery / publishing</td>
  <td class="amount" style="text-align:right;">[PHASE3_AMOUNT]</td>
</tr>
```

Calculate amounts as:
- Phase 1: `total × 0.30`, then subtract initiation fee
- Phase 2: `total × 0.40`
- Phase 3: `total × 0.30`
- Format all values as `€X,XXX.XX`

### Phase contract (single phase):
Only include the row for the current phase. Example for a Phase 2 contract:
```html
<tr>
  <td>Phase 2 — 40%</td>
  <td>Due after written client approval of Phase 2</td>
  <td class="amount" style="text-align:right;">[PHASE2_AMOUNT]</td>
</tr>
```

### Custom agreement:
Use generic milestone labels:
```html
<tr>
  <td>Payment 1 (30%)</td>
  <td>Due before work begins</td>
  <td class="amount" style="text-align:right;">[AMOUNT]</td>
</tr>
<tr>
  <td>Payment 2 (40%)</td>
  <td>Due at agreed midpoint</td>
  <td class="amount" style="text-align:right;">[AMOUNT]</td>
</tr>
<tr>
  <td>Payment 3 (30%)</td>
  <td>Due before final delivery</td>
  <td class="amount" style="text-align:right;">[AMOUNT]</td>
```

### Payment note — initiation fee wording:
- Standard/Phase contract with an initiation fee: `The project initiation fee (€[INIT_FEE]) is non-refundable.`
- Custom agreement with no initiation fee: `No project initiation fee applies to this Custom Agreement.`

## 7. Add-on Rows (`{{ADDON_ROWS}}`)

For each add-on that is **toggled ON** in Step 6, insert one `<tr>` row. For add-ons that are off, do not insert anything.

Add-on row template:
```html
<tr>
  <td class="addon-name">[ADD-ON NAME]<div class="addon-note">[NOTE]</div></td>
  <td class="addon-price">[PRICE]</td>
</tr>
```

## 12. Step 1 Dashboard Field — Dedicated Google Account

The contract wizard Step 1 must include two new fields below the existing client detail fields:

```
Dedicated Google account email:  [input: type=email, placeholder="e.g. clientname@gmail.com"]
Password:                        [input: type=password, placeholder="Store securely"]
```

- These fields are optional at the time of contract creation.
- Both values are stored in the client record in the dashboard (not in the HTML contract itself, except the email).
- Only the email address is injected into the contract via `{{CLIENT_GOOGLE_ACCOUNT}}`.
- The password is stored locally in the dashboard for Cess's reference only and is never written into the contract file.
- If left blank, `{{CLIENT_GOOGLE_ACCOUNT}}` is replaced with `[to be confirmed]` in the output.

## 13. What Copilot Must Not Do

- Do not modify `contract-template.html` — always generate a new output file
- Do not add any framework, library, or build tool dependency
- Do not inline any JavaScript into the contract HTML — it is a static document
- Do not use `position: fixed` anywhere in the contract CSS
- Do not use em dashes (`—`) in the project name or contract type label — use `·` (middle dot) as the separator between the phase and contract type in the subtitle line
- Do not use `Page 1 of 1` — always count actual pages and update `pf-num` accordingly
- Do not center-align any body text — all contract body text is left-aligned
- Do not change any wording in sections 9 through 13 (IP, Confidentiality, Liability, Termination, Governing Law) — these are legally fixed
